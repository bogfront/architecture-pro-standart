Краткое описание выбранной концепции MVP:
1. Вводится выделенный `Сервис депозитных заявок (MVP)` как новый слой между цифровыми каналами и АБС.
2. Сайт и интернет-банк работают с этим сервисом по API, а не напрямую с АБС.
3. Сервис хранит заявки, статусы и журнал уведомлений в собственной БД (`MS SQL`) и интегрируется с АБС через адаптер интеграции.
4. Каталог ставок централизуется в контуре АБС (новая подсистема ведения ставок) с ролевым доступом для бэк-офиса депозитов и кредитов.
5. SMS-уведомления отправляются из сервиса заявок через существующий SMS-шлюз; проверка SMS-кода в интернет-банке остается в существующем ядре без вендорных доработок.
6. Для устойчивости используются асинхронные события смены статусов заявок (через Kafka-совместимый контур интеграции), но интернет-банк взаимодействует только с API сервиса заявок.

Команды, участвующие в реализации:
1. Команда интернет-банка (банк + подрядчик): доработка UI/API интеграции с сервисом заявок.
2. Команда АБС: подсистема ставок, обработка заявок и адаптер интеграции.
3. Команда кол-центра: прием лидов из сайта и отображение статусов.
4. Команда цифровой трансформации: orchestration работ, NFR/SLA, архитектурные контракты.
5. Команда эксплуатации/инфраструктуры: отказоустойчивость между ЦОД, мониторинг, релизы.
### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

1. Прямая интеграция интернет-банка и сайта с АБС.
Плюсы: меньше новых компонентов в MVP.
Минусы: рост нагрузки на перегруженную БД АБС, низкая отказоустойчивость и слабая масштабируемость.  
Статус: отклонено.

2. Полная реализация через доработку монолита интернет-банка (без выделенного сервиса заявок).
Плюсы: меньше операционных контуров.
Минусы: зависимость от вендора ядра, низкая скорость изменений, слабая переиспользуемость для сайта/кол-центра.  
Статус: отклонено.

3. Немедленный полный переход на микросервисную архитектуру интернет-банка.
Плюсы: стратегически правильное направление.
Минусы: не укладывается в сроки/риски MVP, большая стоимость трансформации.  
Статус: отложено (эволюционный переход).

**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

1. Добавление нового сервиса и интеграционного контура увеличивает сложность эксплуатации.
2. АБС остается критическим узким местом (вертикальное масштабирование Oracle).
3. При слабой проработке идемпотентности возможны дубли заявок и уведомлений.
4. Для достижения SLA 99.9% потребуется отдельная проработка DR-плана между ЦОД, что может выйти за рамки MVP.
5. Временная гибридная схема (часть процесса в новом сервисе, часть в АБС) требует строгого управления версиями API и тестирования интеграций.
