Ключевое решение:
1. Вводится `Online Credit Hub` как единая точка приема онлайн-заявок с сайта и интернет-банка.
2. Hub сразу направляет онлайн-заявки в Кредитный конвейер (API/event), чтобы убрать задержку суточного обмена АБС->конвейер для нового канала.
3. Для продолжения в отделении используется единый `Application ID`; АБС получает доступ к данным заявки через адаптер Hub.
4. Предодобренные предложения в интернет-банке строятся на уже рассчитанных данных (витрина предложений), без запуска дополнительных массовых скорингов.
5. Уведомления по статусам идут через централизованный Notification-компонент Hub и SMS-шлюз.

Обоснование:
1. Выполняется бизнес-цель same-day обработки предодобренных кредитов.
2. Снимается ограничение суточного обмена для цифрового канала без перестройки legacy-потока АБС.
3. Ограничивается дополнительная нагрузка на критичную АБС.
4. Сохраняется расширяемость и наблюдаемость процесса на целевой горизонт.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

1. Оставить онлайн-заявки через АБС и текущий суточный обмен с конвейером.
Плюсы: минимальные изменения.
Минусы: не выполняется цель same-day по предодобренным заявкам.
Статус: отклонено.

2. Интегрировать сайт и интернет-банк напрямую с Кредитным конвейером (без Hub).
Плюсы: меньше компонентов.
Минусы: дублирование логики в каналах, сложнее единый audit/идентификатор/уведомления.
Статус: отклонено.

3. Массово пересчитывать предодобрения в скоринге по расписанию.
Плюсы: больше готовых офферов.
Минусы: против ограничения по нагрузке на скоринг в пике.
Статус: отклонено.

**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

1. Новый Hub добавляет операционную сложность и требует 24/7 эксплуатации.
2. Появляется зависимость от стабильности интеграции Hub <-> Кредитный конвейер.
3. При некорректной дедупликации возможны дубли заявок из разных каналов.
4. Сохранится legacy-контур АБС для части операций, поэтому потребуется аккуратная синхронизация статусов.
5. Для соответствия ИБ-требованиям нужны строгие политики хранения и маскирования ПДн в логах.
