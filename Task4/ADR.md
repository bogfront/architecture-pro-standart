### <a name="_u8xz25hbrgql"></a>**Нефункциональные требования**

Ключевые решения:
1. Вводится `Сервис публикации ставок` как отдельный компонент между АБС и потребителями ставок.
2. Источник ставок переносится из XLS в `ABS Deposit Rate Module`; только подтверждённые ставки публикуются наружу.
3. Во внутренний кол-центр ставки передаются через системную интеграцию (API/шина событий внутри банка).
4. Для партнёра используется файловая публикация (CSV/Excel), подписание контрольной суммы, шифрование архива и доставка по согласованному каналу обмена файлами.
5. Реализуется журнал публикаций и ретраи, чтобы выдержать SLA и обеспечить прослеживаемость.

Логика выбора:
1. Выполнено ограничение «без API/SFTP партнёра».
2. Уменьшена ручная работа и риск расхождения ставок.
3. Снята зависимость консультаций КЦ от нестабильных ручных Excel-процессов.
4. Решение укладывается в MVP (6 месяцев) и расширяемо на целевой горизонт (1 год).
### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

1. Передавать ставки партнёру вручную из Excel по email.
Плюсы: быстро стартовать.
Минусы: высокий операционный риск, нет гарантии актуальности и аудита, не масштабируется.
Статус: отклонено.

2. Дать партнёру прямой доступ в систему кол-центра банка.
Плюсы: единый источник данных.
Минусы: риски ИБ и разграничения доступа, организационно сложно, долгое согласование.
Статус: отклонено.

3. Сразу строить B2B API для партнёра.
Плюсы: технологически оптимально в долгую.
Минусы: прямо против текущего ограничения партнёра «только файлы».
Статус: отложено.

**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

1. Файловый обмен с партнёром хуже управляется, чем online API, и требует операционного контроля.
2. Возможны задержки доставки файла по внешним причинам (канал передачи у партнёра).
3. Нужны чёткие регламенты версионирования формата файла; иначе высокий риск ошибок загрузки у партнёра.
4. При недоступности АБС или модуля ставок публикация будет отставать; необходим fallback на последнюю валидную версию.
5. Переход с XLS-процесса потребует обучения сотрудников и изменения регламентов работы бэк-офиса.
