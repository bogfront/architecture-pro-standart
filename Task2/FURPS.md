| Код | Требования | Комментарий |
|-----|------------|-------------|
| F | Функциональные (Functionality) | |
| F | Подача заявки на депозит с сайта для нового клиента: ФИО + телефон, передача заявки в систему кол-центра, последующий обзвон менеджером. | MVP-сценарий привлечения новых клиентов вне интернет-банка. |
| F | Подача заявки на депозит в интернет-банке: выбор продукта, счёта списания и суммы, подтверждение операции по SMS-коду. | Ключевой цифровой канал для действующих клиентов. |
| F | Отображение актуальных ставок по депозитам на сайте и в интернет-банке; в интернет-банке дополнительно персонализированные ставки. | Единый источник ставок нужен для исключения ручных XLS-расчётов. |
| F | В MVP подтверждение условий депозитной заявки выполняет менеджер бэк-офиса в АБС. | Соответствует целевому промежуточному состоянию (6 месяцев). |
| F | SMS-уведомления клиенту о подтверждении ставки и об открытии депозита. | Информирование обязательно по условию задачи; желательно без доработки ядра вендора. |
| F | Подача кредитной заявки с сайта: ФИО + телефон; при наличии паспортных данных — онлайн-предрешение (одобрение/неодобрение) на основе скоринга и БКИ. | Для новых клиентов доступны только данные БКИ и анкеты. |
| F | Подача кредитной заявки в интернет-банке: выбор параметров, подтверждение SMS-кодом, повторный скоринг при отправке заявки. | Для действующих клиентов должен работать сценарий предодобрений. |
| F | Поддержка сквозного кредитного кейса: заявка, начатая на сайте, должна продолжаться в отделении как та же заявка. | Исключает дублирование и ошибки обработки. |
| F | Уведомления по изменению статуса кредитной заявки через SMS. | Требование бизнеса по повышению прозрачности процесса. |
| U | Удобство использования (Usability) | |
| U | Клиентские формы подачи заявок (сайт/интернет-банк) должны содержать минимально необходимое число полей для MVP. | Снижение отказов на этапе заполнения. |
| U | Единый UX/UI в соответствии с корпоративной дизайн-системой и бренд-элементами банка. | Прямое требование бизнеса и маркетинга. |
| U | Пользователь должен получать понятный текущий статус заявки и следующий шаг (например, «ожидайте звонка», «придите в отделение»). | Дополнительное требование; уменьшает нагрузку на кол-центр. |
| R | Надёжность (Reliability) | |
| R | Доступность клиентских цифровых сервисов (сайт/интернет-банк/новые сервисы заявок) не ниже 99.9%. | Бизнес-требование 24/7 и SLA. |
| R | Отказ одного ЦОД не должен останавливать прием заявок; требуется работа через резервный ЦОД. | Сейчас фактически один активный ЦОД, нужен план failover. |
| R | Новые интеграции не должны ухудшать стабильность АБС; исключить прямую синхронную зависимость интернет-банка от API/БД АБС в пике. | БД АБС перегружена и масштабируется только вертикально. |
| R | Гарантированная доставка критичных событий (заявка создана, статус изменен, SMS отправлено) с идемпотентной повторной обработкой. | Дополнительное требование; снижает риск потери заявок при сбоях. |
| P | Производительность (Performance) | |
| P | Время отклика клиентских операций чтения (списки продуктов/ставок, статусы) — p95 не более 300 мс, p99 не более 800 мс. | Конкретизация требования «миллисекунды», устраняет текущую проблему >1 сек. |
| P | Время отклика операций подачи заявки (без внешних проверок) — p95 не более 1.5 сек. | Реалистичная цель для форм с валидацией и записью. |
| P | Заявки по предодобренным кредитам должны попадать в обработку в тот же рабочий день (без ожидания суточного батча из АБС). | Архитектурно требует отдельного канала интеграции, не зависящего от текущего обмена АБС->конвейер 1/сутки. |
| P | Дополнительные процессы не должны увеличивать нагрузку на скоринг в пиковые часы из-за массовых предрасчетов. | Прямое ограничение владельца скоринга. |
| S | Поддерживаемость (Supportability) | |
| S | Архитектура MVP должна быть расширяемой до целевого состояния «без участия бэк-офиса» для депозитов. | Не допускать тупиковой реализации «только под MVP». |
| S | Обязательная техническая документация: контекст/контейнеры, API-контракты, форматы событий, регламенты отказоустойчивости. | Прямой запрос команды трансформации на последующее расширение. |
| S | Наблюдаемость: централизованные логи, бизнес-метрики (конверсия, SLA по этапам), трассировка интеграционных вызовов, алерты. | Дополнительное требование; необходимо для эксплуатации 24/7 и SLA 99.9%. |
| S | Версионирование API и схем событий для безопасных изменений между интернет-банком, сайтом, КЦ, АБС и кредитным контуром. | Дополнительное требование; снижает риск регрессий при развитии. |
| +R | + Ограничения (Restrictions) | |
| +R | Защита чувствительных данных в каналах связи: TLS для сайта и интернет-банка, защищенные внутренние каналы между сервисами. | Прямое требование по шифрованию трафика. |
| +R | Подтверждение критичных операций через SMS-код (депозит/кредит в интернет-банке). | Регуляторика и антифрод-практика банка. |
| +R | Максимально использовать существующий стек и экспертизу банка: Oracle/MS SQL, .NET, Java, Python, PostgreSQL. | Снижение рисков внедрения и стоимости владения. |
| +R | При необходимости очередей использовать Kafka, но учитывать несовместимость текущего монолита интернет-банка с Kafka. | Ограничение на архитектурное решение: нужен промежуточный слой/микросервис. |
| +R | АБС не может масштабироваться горизонтально; новые процессы должны выносить клиентскую нагрузку из АБС и использовать асинхронный обмен. | Критичное инфраструктурное ограничение. |
| +R | Для новых клиентов обязательна очная идентификация в отделении перед выдачей кредитов/финальным открытием продуктов. | Регуляторное/бизнес-ограничение процесса обслуживания. |
